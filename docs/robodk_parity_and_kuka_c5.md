# RoboDK Parity And KUKA C5 Direction

This document updates the project target for future RoboDK-class operation.

## Target

`winding_pose_solver` should become the solver-owned runtime behind a
RoboDK-class winding workbench:

- offline and online IK/FK/path continuity remain in this repository;
- model and visual assets remain tied to solver model identity;
- future animation control, instruction insertion, and program export expose
  stable contracts before implementation details are added;
- `winding-motion-module` stays an adapter facade and never implements a second
  solver, collision engine, or controller post processor.

## Researched Capability Map

RoboDK parity should be split into four capability layers:

1. Visual inspection and playback
   - Load solver-owned model assets.
   - Drive link transforms from solver FK/path results.
   - Support play, pause, stop, seek, step, run-from-instruction, and playback
     speed controls.

2. Program instruction IR
   - Represent motion and non-motion instructions as data.
   - Required instruction kinds: `move_j`, `move_l`, `set_frame`, `set_tool`,
     `set_speed`, `set_rounding`, `set_do`, `wait_di`, `pause`,
     `call_program`, `message`, and `custom_code`.
   - Instruction insertion should happen at the IR/result layer first, then be
     materialized into RoboDK or KUKA output.

3. RoboDK station materialization
   - Existing `src/robodk_runtime` remains the owner for live RoboDK station
     import and program creation.
   - RoboDK-specific calls stay on Windows and should attach to the active
     station.

4. KUKA C5 ecosystem export
   - First target: KUKA KRL `.src/.dat` style export for KR C5 / KSS 8.7.
   - Keep extension points for `mxAutomation` and `SRCI` planning, but do not
     treat them as the same artifact as offline KRL code.
   - Controller-specific syntax should be generated by a post/export layer, not
     by search or IK modules.

## Reserved Interfaces

The following runtime endpoints are reserved:

- `GET /api/capabilities`
- `POST /api/simulation/control`
- `POST /api/program/plan`
- `POST /api/program/export`

Only `GET /api/capabilities` is implemented. The other endpoints return
structured `501` errors until the corresponding runtime layer is built.

The shared program IR lives in `src/core/program_ir.py`. It is intentionally
controller-neutral so the same selected path can later be rendered, simulated,
materialized into RoboDK, or exported for KUKA C5.

## Boundary Rules

- Search, repair, FK, IK, continuity, and future collision decisions are solver
  responsibilities.
- `src/robodk_runtime` owns RoboDK station behavior and final materialization.
- Future KUKA C5 export code should consume selected paths and program IR; it
  must not run a new search or choose IK candidates.
- `winding-motion-module` may expose TypeScript methods for the reserved
  endpoints, but should not emulate them when the solver reports
  `not_implemented`.
